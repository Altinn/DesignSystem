Frontend Guidelines

For å få en felles forståelse av \"best practice\" for frontend utvikling i Altinn har vi samlet noen retningslinjer og regler. På denne måten kan vi sikre frontend-kode av høy kvalitet som fungerer godt og når alle mennesker - ikke bare i dagens nettlesere og enheter, men også i morgendagens.

Tilgjengelighet

Det er viktig gjøre innhold tilgjengelig for alle. God tilgjengelighet kan hjelpe mennesker med funksjonshemninger å oppfatte innholdet vårt på en meningsfull måte. Dette gjelder f.eks blindhet, nedsatt syn og hørsel, lærevansker og begrenset bevegelse. Det er flere ting vi kan gjøre, men det første og viktigste er å skrive semantisk god HTML.

WCAG 2.0-standarden

Forskrift om universell utforming av IKT-løsninger stiller krav om at nettsider må oppfylle 35 av 61 suksesskriterier i standarden Retningslinjer for tilgjengelig webinnhold (WCAG) 2.0. Her finner du oversikt over minimumskravene som er omfattet av forskriften innenfor frontend-området.

Responsive bilder

The goal of responsive images is to deliver the best quality image usable by the accessing device, but nothing more.

This code is backwards compatible. If a browser happens along that doesn’t understand the srcset attribute, it will simply ignore it and render the image according to the original 1993 specification. The "src" is principally used as a default image and for browsers that don’t recognize srcset, whereas srcset contains all of the available high-res format images for that placeholder. When rendering the img element the browser will determine for itself which image file is the most appropriate.(Kilde)

Multiple images

With extra options in the srcset, separated by commas, we’re not telling the browser what image to display, we’re letting it know the options that are available to it, and allowing it to select for itself. The browser will only download one of these images.

  • .sg-... Spesifiserer stilsetting for presentasjon av Pattern Lab

CSS Convention

To ensure readability and consistency, this styleguide has the following convention:

Square brackets denote optional parts
.< moduleName > [ - < componentName > ][ - < state > ] {}

Examples

module container/root
.searchResults {}

components of a module
.searchResults-heading {}

state: such as AJAX loading
.searchResults-isLoading {}

Each of these class names are semantic. Module, component and state are all delimitted by dashes. Each bit is written in lowerCamelCase.
